# Codex 多线程执行方案

## 1. 总体原则

这个项目不要只开一个对话从头做到尾。最有效的方式是先用一个“主控对话”统一架构和规则，再开多个“专项对话”并行推进。每个专项对话只负责一个边界清晰的模块，最后把产出合并回主控对话。

核心原则：

1. 主控对话负责方向、架构、接口和合并，不负责所有细节开发。
2. 专项对话只做一个模块，避免跨模块乱改。
3. 每个专项对话开始时都必须读取项目 PRD 和本执行方案。
4. 所有专项对话的产出都要写入项目目录的文档或代码，不能只停留在聊天里。
5. 涉及接口、数据结构、目录结构的变更，必须先回到主控对话确认。
6. 新增策略、因子、数据源时，要优先遵守插件化架构，不要写死在主流程里。

## 2. 推荐对话划分

建议至少分 6 条 Codex 对话并行推进。

### 2.1 主控对话：项目总架构与合并

职责：

- 维护 PRD
- 维护整体架构
- 确定模块边界
- 确定接口规范
- 合并各专项对话产出
- 处理跨模块冲突
- 决定下一阶段优先级

不做：

- 不在主控对话里写大量具体因子代码
- 不在主控对话里反复调试前端细节
- 不在主控对话里临时改一堆策略逻辑

主控对话的推荐开场：

```text
你是这个 AI 量化选股平台项目的主控架构师。
请先读取当前目录下的：
1. AI量化选股平台_PRD与实施计划.md
2. Codex多线程执行方案.md

你的任务是维护项目总架构、模块边界、接口规范和各专项任务的合并，不要直接把所有功能都堆在一个文件里。
接下来请先检查当前项目文件，给出本阶段最应该推进的任务清单。
```

### 2.2 数据线程：免费数据源与数据管道

- 实现 A股、港股基础行情获取
- 实现交易日历、指数数据、基础财务和估值数据获取
- 设计本地数据存储结构
- 做数据清洗和字段标准化
- 做数据质量检查
- 产出数据字段字典

主要产出：

- `docs/数据字典.md`
- `docs/数据源评估.md`
- `backend/data_sources/`
- `backend/data_pipeline/`
- `data/` 本地数据目录

数据线程推荐开场：

```text

请先读取：
1. AI量化选股平台_PRD与实施计划.md

只负责免费数据源、A股/港股数据管道、数据清洗、字段标准化和数据质量检查。
不要实现策略、前端页面或 AI 模型。

请先输出数据源评估、数据字段规范和本地存储方案，然后再开始写代码。
```

- 设计解释模板接口

- `docs/策略插件规范.md`

## 3. 串行与并行关系

### 3.1 必须先串行完成

以下内容必须先由主控对话完成，完成前不建议大规模并行写代码：

1. 项目目录结构
2. 数据字段命名规范
3. 策略插件接口
4. 统一信号输出格式
5. API 基础规范
6. 前后端技术栈选择

### 3.2 可以立即并行

主控对话确定基础规范后，可以同时开：

- 数据线程
- 策略框架线程
- 前端体验线程
- 后端平台线程

这四条线互相依赖较少，可以最快推进。

### 3.3 需要稍后启动

回测线程可以在策略信号格式初步确定后启动。

AI 模型线程建议最后启动，至少要等：

- 数据管道初步可用
- 因子表结构确定
- 策略信号格式确定
- 回测规则确定

## 4. 推荐推进顺序

### 阶段 0：主控准备

目标：把所有线程能共同遵守的规则先定下来。

任务：

- 创建项目目录结构
- 创建 `docs/`
- 创建 `backend/`
- 创建 `frontend/`
- 写 `docs/架构总览.md`
- 写 `docs/策略插件规范.md`
- 写 `docs/API接口规范.md` 初稿
- 写 `docs/数据字典.md` 初稿

完成标准：

- 每个专项线程都知道自己该读哪些文件
- 每个专项线程都知道不能改哪些模块
- 新增策略的接口边界清楚

### 阶段 1：四线并行

#### 数据线程

目标：跑通免费数据。

任务：

- 评估 AkShare 可用接口
- 拉取 A股主流股票日线
- 拉取港股主流股票日线
- 标准化字段
- 存入 Parquet / DuckDB
- 生成数据质量报告

#### 策略框架线程

目标：跑通策略插件机制。

任务：

- 实现策略配置读取
- 实现策略注册表
- 实现统一信号输出结构
- 实现一个示例策略
- 写策略新增说明

#### 前端体验线程

目标：先用 mock 数据做出完整小白界面。

任务：

- 首页
- A股榜单
- 港股榜单
- 股票详情
- 观察名单
- 报告中心
- 风险等级展示

#### 后端平台线程

目标：先用 mock 数据提供 API。

任务：

- FastAPI 项目结构
- 登录接口
- 用户角色
- 榜单接口
- 股票详情接口
- 报告接口
- 任务状态接口

### 阶段 2：第一次集成

目标：把 mock 链路替换成真实链路的一部分。

任务：

- 数据线程产出的股票池接入策略框架
- 策略框架产出的评分接入后端 API
- 前端榜单页改读后端 API
- 后端任务状态展示真实数据更新时间

完成标准：

- 本地打开平台，可以看到真实或半真实的 A股/港股候选榜单
- 用户可以登录
- 用户可以加入观察名单
- 管理员可以看到数据更新状态

### 阶段 3：回测与可信度

目标：让榜单结果具备基本可信度。

任务：

- 实现日频回测
- 实现周频回测
- 加入交易约束
- 输出收益曲线、最大回撤、胜率、换手率
- 在前端展示回测结果
- 标记回测风险和数据限制

完成标准：

- 每个策略都有基础回测结果
- 榜单不是只给分数，还能看到历史表现

### 阶段 4：AI 模型

目标：在已有规则策略基础上增加 AI 排序策略。

任务：

- 生成训练样本
- 定义标签
- 训练 LightGBM
- 输出 AI 评分
- 接入策略插件
- 做 AI 策略回测
- 生成小白解释

完成标准：

- AI 策略和规则策略可以同时存在
- 用户可以切换查看不同策略结果
- 新增 AI 策略不破坏原有主流程

### 阶段 5：体验与稳定性

目标：内部可试用。

任务：

- 优化页面文案
- 完善报告中心
- 完善权限
- 完善任务日志
- 完善异常提示
- 写本地部署说明
- 写日常使用手册

## 6. 每个线程完成后的汇报格式

每个专项对话完成阶段任务后，要把结果用下面格式交给主控对话。

```text
线程名称：
本轮完成：
新增/修改文件：
对其他线程的影响：
需要主控确认的问题：
下一步建议：
```

示例：

```text
线程名称：策略框架线程
本轮完成：实现 strategy.yaml 规范、策略注册表、统一信号输出结构
新增/修改文件：
- docs/策略插件规范.md
- backend/strategies/registry.py
- backend/strategies/base.py
对其他线程的影响：
- 回测线程需要按 StrategySignal 格式读取信号
- 后端线程可以从 scoring_results 表读取榜单结果
需要主控确认的问题：
- risk_score 是否统一用 0-100 分
下一步建议：
- 接入第一个规则策略 momentum_quality_daily
```

## 8. 合并规则

1. 专项线程不要随意修改其他线程负责的目录。
2. 如果必须修改公共接口，先写到“需要主控确认的问题”，不要直接改。
3. 主控对话定期检查所有文件，统一处理冲突。
4. 每次阶段集成前，先跑通本地启动和基础测试。
5. 文档和代码不一致时，以主控对话更新后的文档为准。
