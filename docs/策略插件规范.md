# 策略插件规范

## 1. 目标

策略必须插件化。新增一个策略时，优先只新增一个策略目录、一个 `strategy.yaml` 和策略实现文件，不修改数据更新、回测、榜单、报告或 API 主流程。

第一版策略框架只负责：

1. 接收已经标准化的因子数据。
2. 按策略逻辑生成统一策略信号。
3. 输出小白可读的入选理由和风险提示字段。
4. 为回测、评分榜单、报告和 API 提供稳定的数据结构。

策略层不负责接真实数据源、不负责前端展示、不负责登录权限、不负责交易执行。

## 2. 推荐目录

```text
backend/strategies/
  base.py
  config.py
  explanations.py
  registry.py
  signals.py
  examples/
    momentum_quality_daily/
      strategy.yaml
      signal.py

backend/factors/
  base.py

backend/scoring/
  base.py
```

## 3. `strategy.yaml` 标准

每个策略目录必须包含一个 `strategy.yaml`。主流程后续应只扫描配置文件，再通过注册机制加载策略类。

### 3.1 必填字段

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `strategy_id` | string | 全局唯一策略 ID，使用英文 snake_case |
| `name` | string | 中文展示名 |
| `market` | list[string] | 支持市场，当前只允许 `cn`、`hk` |
| `frequency` | string | `daily` 或 `weekly` |
| `universe` | string | 股票池名称，例如 `mainstream` |
| `factors` | list[string] | 策略依赖的因子名称 |
| `model.type` | string | `rule`、`ml_ranker`、`ml_regressor` 等 |
| `horizon` | list[string] | 观察周期，例如 `5d`、`20d`、`4w` |
| `rebalance_rule` | string | 调仓/更新节奏，例如 `daily_after_close` |
| `risk_rule` | object | 策略侧风险过滤和风险评分参数 |
| `explanation_template` | string | 解释模板 ID |
| `entrypoint.module` | string | Python 模块路径 |
| `entrypoint.class` | string | 策略类名，必须继承 `BaseStrategy` |

### 3.2 推荐字段

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `version` | string | 策略版本 |
| `description` | string | 策略说明 |
| `author` | string | 负责人或线程 |
| `enabled` | bool | 是否默认启用 |
| `tags` | list[string] | 标签，例如 `momentum`、`quality` |
| `score_range` | object | 分数范围，默认 `0-100` |
| `output_top_n` | list[int] | 建议输出数量，例如 `20`、`50` |

### 3.3 示例

```yaml
strategy_id: momentum_quality_daily
name: 趋势质量日频策略
version: 0.1.0
description: 使用趋势、质量、量能和波动风险因子生成 A股/港股日频候选股信号。
enabled: true
market:
  - cn
  - hk
frequency: daily
universe: mainstream
factors:
  - momentum_20d
  - momentum_60d
  - quality_score
  - amount_change_20d
  - volatility_20d
model:
  type: rule
horizon:
  - 5d
  - 20d
rebalance_rule: daily_after_close
risk_rule:
  exclude_st: true
  exclude_suspended: true
  min_avg_amount_20d: 50000000
  max_risk_score: 85
explanation_template: default_stock_selection
entrypoint:
  module: backend.strategies.examples.momentum_quality_daily.signal
  class: MomentumQualityDailyStrategy
output_top_n:
  - 20
  - 50
tags:
  - momentum
  - quality
```

## 4. 策略注册机制

策略注册分三步：

1. 扫描策略目录下的 `strategy.yaml`。
2. 校验配置字段，并读取 `entrypoint.module` 与 `entrypoint.class`。
3. 将策略类和配置注册到 `StrategyRegistry`，主流程通过 `strategy_id` 获取策略实例。

主流程建议只依赖以下接口：

```python
registry = StrategyRegistry()
registry.discover("backend/strategies/examples")
strategy = registry.create("momentum_quality_daily")
signals = strategy.generate_signals(factor_data, context)
```

后续新增策略时，不应把策略 ID 写死进数据管道、API、前端或回测引擎。

## 5. 策略抽象接口

所有策略类必须继承 `BaseStrategy`，并实现：

```python
generate_signals(factor_data, context) -> list[StrategySignal]
```

### 5.1 输入约定

策略输入只能是上游已经准备好的因子数据，不能在策略内部调用外部数据源。

允许两种输入形态：

1. 标准长表：字段来自 `factor_values`，包括 `trade_date`、`symbol`、`market`、`factor_name`、`factor_value`、`factor_score`。
2. 宽表/mock 数据：每行包含 `trade_date`、`symbol`、`market`，并把因子名作为列名，例如 `momentum_20d`。

策略应优先使用 `factor_score`，分数范围建议为 `0-100`。缺失因子可使用中性分 `50`，但必须在解释或风险字段中体现不确定性。

### 5.2 上下文约定

`StrategyContext` 至少包含：

| 字段 | 说明 |
| --- | --- |
| `trade_date` | 信号日期 |
| `frequency` | `daily` 或 `weekly` |
| `market_scope` | 本次运行的市场范围 |
| `universe` | 股票池 |
| `metadata` | 扩展信息 |

## 6. 统一策略信号

每个策略最终必须输出 `StrategySignal`，并可序列化为下表字段。

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `trade_date` | string | 是 | 信号日期，建议 `YYYY-MM-DD` |
| `strategy_id` | string | 是 | 策略 ID |
| `symbol` | string | 是 | 内部统一股票代码 |
| `market` | string | 是 | `cn` 或 `hk` |
| `score` | float | 是 | 综合分数，`0-100`，越高越值得观察 |
| `rank` | int | 是 | 同策略、同市场内排名，从 1 开始 |
| `risk_score` | float | 是 | 风险分数，`0-100`，越高风险越大 |
| `risk_level` | string | 是 | `低`、`中`、`中高`、`高` |
| `horizon` | string | 是 | 观察周期，例如 `5d`、`20d`、`4w` |
| `reason_summary` | string | 是 | 小白可读的入选理由摘要 |
| `positive_factors` | list[object] | 是 | 主要正向因素 |
| `negative_factors` | list[object] | 是 | 主要风险因素 |
| `extra` | object | 否 | 扩展字段，不能替代标准字段 |

### 6.1 因子贡献格式

`positive_factors` 和 `negative_factors` 的单项格式：

```json
{
  "factor_name": "momentum_20d",
  "display_name": "20日趋势",
  "value": 0.12,
  "score": 78.5,
  "weight": 0.35,
  "direction": "positive",
  "explanation": "近20日趋势强于多数同市场股票"
}
```

## 7. 因子接口

因子模块只定义可复用因子计算接口，不绑定具体策略。

`BaseFactor` 必须提供：

```python
calculate(input_data, context) -> list[FactorValue]
```

`FactorValue` 输出字段应与 `docs/数据字典.md` 的 `factor_values` 保持一致：

- `trade_date`
- `symbol`
- `market`
- `factor_name`
- `factor_value`
- `factor_score`
- `metadata`

因子模块可以后续由数据线程或模型线程填充真实计算逻辑，但策略框架线程不直接接数据源。

## 8. 评分接口

评分模块负责把策略信号整理成榜单输出，不重新计算因子。

第一版评分输出建议包含：

| 字段 | 说明 |
| --- | --- |
| `trade_date` | 榜单日期 |
| `strategy_id` | 策略 ID |
| `frequency` | 日频/周频 |
| `market` | 市场 |
| `top_n` | 榜单数量 |
| `candidates` | 候选股信号列表 |
| `generated_at` | 生成时间 |

评分层可以做 Top 20 / Top 50 截取、市场内排序、风险等级过滤，但不应修改原始策略逻辑。

## 9. 解释模板接口

解释模板负责把策略信号里的因子分数、风险分数和贡献项转成小白可读文案。策略实现可以选择模板，但不应在 API 或前端重复拼接专业解释。

第一版接口位于 `backend/strategies/explanations.py`，包含：

| 接口 | 说明 |
| --- | --- |
| `ExplanationContext` | 解释模板输入上下文，包含策略 ID、股票代码、市场、因子分数、风险分数和贡献项 |
| `BaseExplanationTemplate` | 解释模板基类，必须实现 `render_reason_summary` |
| `DefaultStockSelectionTemplate` | 默认选股解释模板，对应 `strategy.yaml` 的 `default_stock_selection` |
| `ExplanationTemplateRegistry` | 解释模板注册表，后续新增模板时不改策略主流程 |

示例策略应通过 `explanation_template` 字段选择模板，并把生成结果写入：

- `reason_summary`：前端和报告优先展示的入选理由。
- `extra.risk_note`：更细的风险说明，后续报告线程可选择展示。

## 10. 新增策略步骤

新增策略时按以下步骤执行，避免把某个策略写死进主流程：

1. 在 `backend/strategies/examples/` 或后续正式策略目录下新增一个策略目录。
2. 新增 `strategy.yaml`，填写 `strategy_id`、`market`、`frequency`、`factors`、`entrypoint` 和 `explanation_template`。
3. 新增策略实现文件，策略类继承 `BaseStrategy`，只实现 `generate_signals`，不直接接数据源。
4. 输入只使用标准化因子数据；如存在缺失因子，使用中性分或跳过样本，并在解释或风险字段中体现不确定性。
5. 输出必须是 `StrategySignal`，且 `rank` 按“同策略、同市场内排名”从 1 开始。
6. 如需要新的解释文案，新增 `BaseExplanationTemplate` 子类并注册模板 ID，不在前端或 API 里拼策略专属文案。
7. 用 `StrategyRegistry.discover(...)` 验证配置可被扫描，用 `SimpleTopNScoringEngine` 验证榜单输出。
8. 补充或更新测试，至少覆盖配置读取、注册、信号生成和 Top N 输出。

最小验证代码：

```python
registry = StrategyRegistry()
registry.discover("backend/strategies/examples")
strategy = registry.create("momentum_quality_daily")
signals = strategy.generate_signals(factor_rows, context)
rankings = SimpleTopNScoringEngine().build_rankings(signals, context.frequency, [20, 50])
```

## 11. 回测与后端接入约定

回测线程和后端平台线程只接收策略框架的稳定输出，不直接依赖具体策略类：

1. 回测线程输入：`StrategySignal` 或 `StrategySignal.to_dict()` 的结果。
2. 后端榜单输入：`RankingOutput.to_dict()` 的结果。
3. 前端展示字段：优先使用 `score`、`rank`、`risk_level`、`reason_summary`、`positive_factors`、`negative_factors`。
4. 策略配置变更：通过 `strategy.yaml` 变更，主流程只重新 discover，不改 API 路由。

## 12. 策略层禁止事项

1. 不直接调用 AkShare、Tushare 或其他外部数据源。
2. 不直接读写 `data/` 本地数据目录。
3. 不直接写前端页面。
4. 不直接处理登录和权限。
5. 不绕过统一回测接口。
6. 不输出“买入、卖出、荐股”等强交易导向字段。
7. 不把某个策略逻辑写进主流程。

## 13. 跨线程接口建议

以下是对其他线程的接口建议，不由策略框架线程直接修改：

1. 数据线程：因子表建议输出 `factor_score`，范围统一为 `0-100`。
2. 回测线程：只接收 `StrategySignal` 或其字典形式，不依赖具体策略类。
3. 后端平台线程：榜单 API 建议读取评分层 `RankingOutput`，不要直接拼策略内部字段。
4. 前端线程：页面文案使用 `reason_summary`、`positive_factors`、`negative_factors` 和 `risk_level`，不要自行推导专业结论。
